## 서블릿 간의 통신과 JSP의 동작 원리

### 1. 서블릿 통신의 필요성
하나의 서블릿이 모든 책임을 지는 대신, 작업을 분리하여 다른 서블릿으로 데이터를 전달하고 처리를 위임해야 하는 상황이 빈번하게 발생한다. <br> <br> 이때 개발자가 직접 대상 서블릿의 인스턴스를 생성할 수는 없다. **서블릿 객체의 생성과 관리 권한은 전적으로 WAS에 있기 때문**이다. 따라서 다른 서블릿을 호출하려면 WAS에게 실행을 위임하는 통신 방식을 사용해야 한다.

### 2. 흐름 제어 방식
서버 내부에서 흐름을 제어하는 방식은 크게 세 가지로 나뉜다.

* **Forward** 
  * 현재 서블릿이 클라이언트의 요청(`Request`) 정보를 그대로 유지한 채 다른 서블릿이나 화면으로 제어권을 완전히 넘기는 방식이다.

* **Include** 
  * 다른 서블릿의 실행 결과를 현재 서블릿의 결과에 포함시키는 방식이다.

* **Redirect** 
  * 클라이언트(브라우저)에게 특정 주소로 다시 요청하라고 지시하는 방식이다. 
  * 요청 흐름이 한 번 끊어졌다가 새롭게 연결된다.

### 3. Response Buffer와 제어권 전환
포워드 방식을 사용할 때 반드시 알아야 할 개념이 **Response Buffer**다. 웹 서버는 응답 데이터를 즉시 전송하지 않고 버퍼라는 임시 저장소에 모았다가 꽉 차면 일괄적으로 전송(Flush)한다. 

>A 서블릿이 B 서블릿으로 제어권을 넘기게 되면, 서버는 응답의 일관성을 위해 A 서블릿이 버퍼에 임시로 작성해 둔 데이터를 모두 비워버린다. B 서블릿이 작성한 새로운 데이터만 클라이언트에게 안전하게 담기기 위함이다. 만약 A 서블릿의 버퍼가 이미 가득 차서 클라이언트에게 일부 데이터가 전송(Flush)되어 버린 후라면, 제어권을 넘길 수 없어 에러가 발생한다.

### 4. JSP (Java Server Pages)
자바 코드 안에 HTML 태그를 일일이 작성해야 했던 서블릿의 불편함을 해소하기 위해 등장한 뷰 템플릿 기술이 JSP다. JSP는 내부적으로 네 가지 주요 태그를 사용하여 자바 코드를 표현한다.

* **지시자(Directive) `<%@ %>`:** 페이지 속성이나 임포트 구문을 정의한다.
* **선언부(Declaration) `<%! %>`:** 멤버 변수나 메서드를 선언한다.
* **스크립틀릿(Scriptlet) `<% %>`:** 일반적인 자바 로직을 작성한다.
* **표현식(Expression) `<%= %>`:** 변수의 값이나 메서드의 반환값을 화면에 출력한다.

사용자가 JSP 페이지를 요청하면 서버(Tomcat)는 이를 내부적으로 자바 서블릿(`.java`) 파일로 변환하여 실행한다. 

결국 서블릿에서 화면 출력을 위해 JSP를 호출하는 작업은 **서블릿에서 또 다른 서블릿을 호출하는 포워드 처리 과정**과 완전히 동일한 원리로 동작한다.
