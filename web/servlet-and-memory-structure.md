## 서블릿의 멀티 스레딩과 안전한 요청 처리

### 1. 자바 스레드와 메모리 구조의 이해
자바 애플리케이션이 실행될 때 스레드가 생성되면 메모리 상에 독립적인 영역이 할당된다.

<img src="https://miro.medium.com/v2/resize:fit:720/format:webp/1*sG2wIZg7SqyhKMKD1jxM9A.png">

* **독립적인 스택 영역** 
  * 스레드가 생성될 때마다 해당 스레드만을 위한 전용 공간인 스택이 할당된다. 

  * 스택에는 메서드 호출 정보와 지역 변수가 저장된다.

* **독립적인 함수 실행** 
  * 각 스레드는 자신만의 스택 위에서 코드를 실행하므로 다른 스레드의 지역 변수를 침범할 수 없다. 

* **Context Switching** 
  * CPU 코어는 한정되어 있으므로 여러 스레드를 아주 빠르게 번갈아 가며 실행한다. 
  * 실행 상태를 저장하고 교체하는 과정을 컨텍스트 스위칭이라 부른다.
<br><br>

### 2. WAS의 멀티 스레드 관리와 서블릿
WAS는 클라이언트의 요청이 들어올 때마다 매번 새로운 스레드를 생성하지 않고 **스레드 풀** 방식을 사용한다. <br> 서버 구동 시 일정 개수의 스레드를 미리 만들어두고, 요청이 올 때마다 빌려준 뒤 다시 반환받는 구조다. 

<img src = "https://hahava.github.io/assets/img/servlet-thread.png"><br>

>이러한 환경에서 서블릿 컨테이너(WAS)는 메모리 효율을 위해 개별 **서블릿 객체를 힙 메모리 영역에 단 하나만 생성**하는 방식으로 관리한다. 따라서 **하나의 서블릿 객체에 수많은 스레드가 동시에 접근하여 실행하는 구조**가 된다.
<br><br>

### 3. 멀티 스레드 환경의 위험성과 Thread-safe 설계
객체가 하나뿐이므로 여러 스레드가 동시에 공유 자원에 접근할 때 치명적인 문제가 발생할 수 있다.

* **지역 변수:** 스택 영역에 저장되며 스레드별로 독립적이므로 안전하다.
* **멤버 변수(Field):** 힙 영역에 저장되어 **모든 스레드가 공유**하므로 데이터가 덮어씌워지는 경쟁 상태가 발생할 수 있다.

공유 자원을 안전하게 관리하기 위해 자바는 임계 구역에 잠금장치를 거는 동기화(`synchronized`) 키워드나 스레드 간 통신 메서드(`wait & notify`)를 제공한다. <br><br>
이는 운영체제의 뮤텍스(Mutex)나 세마포어(Semaphore)와 유사한 상호 배제 원리다. 하지만 동기화를 남발하면 서버의 처리 속도가 급격히 저하되는 병목 현상이 발생한다.

따라서 서블릿을 설계할 때는 상태를 유지하는 멤버 변수 사용을 극도로 제한하고, 지역 변수나 요청 객체(`HttpServletRequest`)를 활용하는 스레드 안전(Thread-safe) 설계를 지향해야 한다.
