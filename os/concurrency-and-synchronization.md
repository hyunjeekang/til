# 동시성 제어와 Thread Safe

## 1. 핵심 개념

#### Thread-safe (스레드 안전성)
여러 스레드가 동시에 하나의 공유 자원에 접근해도 프로그램이 의도한 대로 정확하게 동작하는 상태를 뜻한다. 우리가 멀티 스레드 환경에서 달성하고자 하는 최종 목표!

#### Race Condition (경쟁 상태)
두 개 이상의 스레드가 공유 자원에 동시에 접근하여 그 값을 변경하려고 할 때, 타이밍에 따라 결과값이 달라지거나 데이터가 훼손되는 버그를 말한다. (스레드 안정성 만족 x)

#### Synchronization (동기화)
이러한 경쟁 상태를 예방하고 스레드들이 자원을 사용하는 순서나 타이밍을 조절하는 전체적인 전략을 동기화라고 부른다.

## 2. 경쟁 상태는 왜 발생하는가?
스레드가 변수를 단순히 읽기만 할 때는 문제가 없다. 하지만 값을 수정하는 작업이 겹칠 때 문제가 생긴다.

우리가 코드를 짤 때 `count = count + 1`이라고 한 줄로 적지만, CPU 내부에서는 이 작업이 3단계로 쪼개져서 실행된다.

1. 읽기: 메모리에서 현재 count 값을 가져온다.
2. 수정: 가져온 값에 1을 더한다.
3. 쓰기: 더해진 새로운 값을 다시 메모리에 덮어쓴다.

만약 count 변수의 값이 0일 때 스레드 A와 B가 동시에 이 코드를 실행하면 다음과 같은 경쟁 상태가 발생할 수 있다.

1. A가 메모리에서 0을 읽어온다.
2. 동시에 B도 메모리에서 0을 읽어온다.
3. A가 0에 1을 더해서 1을 메모리에 저장한다.
4. B도 자신이 아까 읽어둔 0에 1을 더해서 1을 메모리에 저장한다.

분명히 1을 더하는 작업을 두 번 실행했지만, 최종 값은 2가 아니라 1이 되어버린다. 수정 결과를 덮어쓰는 과정에서 데이터가 날아간 것이다. 

## 3. Lock 기반 동기화
위와 같은 문제를 막기 위해 특정 코드가 실행되는 동안 다른 스레드가 자원에 접근하지 못하도록 잠그는 기술을 사용한다. 대표적으로 뮤텍스와 세마포어가 있다.

* **Mutex (뮤텍스)**
  접근 권한이 딱 1개인 방식이다. 자원에 접근하기 위해 락을 획득한 스레드만이 작업을 마친 후 락을 해제할 수 있다. 소유권이 엄격하게 지켜진다.

* **Semaphore (세마포어)**
  접근 권한이 여러 개 존재하는 방식이다. 설정한 개수만큼 여러 스레드가 동시에 자원에 접근할 수 있다. 락을 획득한 스레드가 아니더라도 다른 스레드가 락을 해제해 줄 수 있어 소유권 개념이 없다.

## 4. Lock-Free 및 최적화 기법
락을 사용하면 한 스레드가 자원을 점유할 때 다른 스레드는 대기 상태로 전환되며, 다시 실행 상태로 돌아오는 과정에서 컨텍스트 스위칭 오버헤드가 크게 발생한다. 이는 프로그램 성능 저하의 원인이 된다. 

이를 해결하기 위해 현대 프로그래밍에서는 락을 쓰지 않거나 최소화하는 다음과 같은 기법들을 추가로 사용한다.

* **원자적 변수와 CAS 알고리즘** <br>
  락을 걸지 않고 하드웨어 수준의 명령어를 사용하여 아주 빠르게 값을 변경한다. 변경을 시도할 때 다른 스레드가 이미 값을 바꿨다면 쿨하게 처음부터 다시 시도하는 방식이다.

* **가시성 제어 (volatile 키워드)**<br>
  각 CPU 코어가 캐시 메모리에 데이터를 복사해 두고 써서 값이 달라지는 것을 막기 위해, 모든 스레드가 항상 메인 메모리에서 직접 데이터를 읽고 쓰도록 강제한다.

* **불변성 (Immutability)**<br>
  객체를 한 번 생성하면 상태를 아예 수정할 수 없게 설계한다. 값을 바꿀 수 없으므로 여러 스레드가 동시에 접근해도 절대 경쟁 상태가 발생하지 않는다.

* **스레드 로컬 (Thread Local)**<br>
  공유 자원을 사용하는 대신, 각 스레드에게 절대 침범받지 않는 독립적인 개인 전용 메모리 공간을 할당하여 충돌 자체를 원천 차단한다.
